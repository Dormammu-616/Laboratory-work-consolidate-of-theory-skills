## Техническое задание лабораторной работы №1
### Цель и теоретическая база
Основная цель лабораторной работы заключается в практическом применении принципов инкапсуляции для защиты внутреннего состояния бизнес-объектов. Проект опирается на концепции сокрытия данных, использование модификаторов доступа, а также создание надежных методов-геттеров и сеттеров. Это позволяет гарантировать сохранение инвариантов класса на протяжении всего жизненного цикла объекта.

### Структура исходного кода
Программа разделена на заголовочные файлы и файлы реализации в соответствии со стандартами C++17.
- `Account.h`: Содержит объявления базового банковского счета и его интерфейса. 
- `Account.cpp`: Включает реализацию основной бизнес-логики и проверки инвариантов.
- `CorporateAccount.h`: Описывает структуру класса-наследника для корпоративных клиентов.
- `CorporateAccount.cpp`: Реализует расширенную функциональность с доступом к защищенным полям.
- `main.cpp`: Точка входа, содержащая тестовые сценарии и демонстрацию работы.

### Защита состояния объекта
В проекте реализован строгий контроль за изменениями внутренних данных, чтобы исключить возможность перевода программы в некорректное состояние.
- Все ключевые атрибуты класса объявлены как `private` для предотвращения прямого изменения из внешнего кода.
- Методы изменения состояния гарантируют, что баланс счета всегда остается неотрицательным (≥0≥0), а имя владельца не является пустой строкой.
- Транзакционные функции предварительно проверяют статус аккаунта и блокируют операции, если счет деактивирован.
- Модификатор `protected` позволяет `CorporateAccount` безопасно применять специфичные правила (например, начисление бонусов или учет комиссий) без нарушения общей инкапсуляции.

### Результаты выполнения работы
Разработанная архитектура успешно защищает объект от неконтролируемых модификаций. Тестовые сценарии подтверждают, что попытки списать средства сверх лимита или провести операцию по замороженному счету корректно отклоняются с выводом соответствующих диагностических сообщений. Применение инкапсуляции сделало интерфейс работы со счетом интуитивно понятным, надежным и готовым к безопасному расширению.

## Техническое задание лабораторной работы №2
### Цель и теоретическая база
Основная цель лабораторной работы заключается в проектировании классов с чётким разделением интерфейса и реализации для обеспечения стабильности публичного API. Проект опирается на идиому Pimpl (Pointer to Implementation), которая позволяет полностью скрыть физическую структуру данных от внешнего пользователя и минимизировать связность кода. Также работа демонстрирует практическое применение внутриклассовых инициализаторов и методов жесткого контроля инвариантов на протяжении всего жизненного цикла объекта.

### Структура исходного кода
Программа разделена на заголовочные файлы и файлы реализации для строгого разграничения интерфейса и скрытой бизнес-логики.
- `UserProfile.h`: Содержит исключительно публичный контракт класса, forward-декларацию структуры `Impl` и умный указатель `std::unique_ptr` для связи с реализацией.
- `UserProfile.cpp`: Включает полное описание приватной структуры `Impl` с полями данных, а также реализацию методов-геттеров, сеттеров и внутренней функции валидации.
- `lab 2 - idiom PIMPL.h` (и соответствующий `.cpp`): Точка входа для тестов (`run_lab`), содержащая сценарии создания профилей и проверки системы на устойчивость к некорректным данным.

### Защита состояния объекта
В проекте реализована двухуровневая защита данных: сокрытие структуры на уровне компиляции и защита логики на уровне выполнения.
- Физические поля (`name`, `age`, `email`) и приватный метод `is_valid_email` инкапсулированы внутри структуры `Impl` в `.cpp` файле, что исключает к ним прямой доступ из внешнего кода.
- Методы изменения состояния предварительно проверяют входящие параметры, гарантируя, что возраст всегда больше нуля, имя не пустует, а электронная почта содержит символ `@`.
- Внутриклассовые инициализаторы исключают появление «мусорных» значений в памяти при создании профиля.
- Любые попытки передать невалидные данные немедленно прерываются генерацией исключения `std::invalid_argument`.

### Результаты выполнения работы
Разработанная архитектура обеспечивает нулевую зависимость клиентского кода от внутренней реализации класса. Изменение состава полей или логики проверки почты в `UserProfile.cpp` больше не требует перекомпиляции файлов, использующих `UserProfile.h`. Тестовые сценарии успешно подтверждают, что профиль невозможно перевести в некорректное состояние, а API остается интуитивно понятным, стабильным и защищенным.

## Техническое задание лабораторной работы №3
### Цель и теоретическая база
Основная цель лабораторной работы — освоить продвинутые механизмы контроля доступа в C++, позволяющие проектировать взаимодействующие классы без нарушения инкапсуляции. Работа демонстрирует применение ключевого слова `friend` для делегирования прав на чтение приватных данных внешнему классу-аналитику, а также использование модификатора `protected` для предоставления части внутреннего функционала классам-наследникам. Важным аспектом работы является обеспечение баланса: мы открываем доступ ровно в той мере, в какой это необходимо, сохраняя при этом публичный API стабильным.

### Структура исходного кода
Проект состоит из набора заголовочных файлов и файлов реализации, каждый из которых отвечает за свой компонент:
- `LogManager.h` и `LogManager.cpp`: Содержат базовый класс управления логами. Приватная структура `Logs` и контейнер для её хранения закрыты от внешнего мира, но класс объявляет `friend class LogInspector`. При добавлении логов используется алгоритм `std::upper_bound` для автоматической сортировки записей по времени.
- `LogInspector.h` и `LogInspector.cpp`: Класс-аналитик, который благодаря дружественным связям может напрямую читать приватный вектор логов. Он вычисляет частоту появления различных кодов ошибок с использованием ассоциативного контейнера `std::map`.
- `ExtendedLogManager.h` и `ExtendedLogManager.cpp`: Класс-наследник `LogManager`, демонстрирующий работу защищенного наследования. Он использует метод `last_type_log()`, доступный через `protected`, для расширения логики базового класса.
- `lab 3 - Stable Public API.cpp` (и `.h`): Клиентский код с тестовыми сценариями, подтверждающий корректность работы API и невозможность несанкционированного изменения данных.

### Защита состояния объекта
Архитектура проекта выстроена таким образом, чтобы исключить неконтролируемые модификации, при этом обеспечив эффективное взаимодействие компонентов:
- Структура данных `Logs` и сам вектор записей `vec_logs` скрыты в `private` секции `LogManager`. Клиентский код может лишь добавлять записи через публичный метод `add_log` и очищать их через `clear_log`, но не может менять уже добавленные логи.
- Класс `LogInspector` получает данные по константной ссылке (`const LogManager&`), что гарантирует безопасность инвариантов `LogManager` — аналитик может читать вектор `vec_logs` как «друг», но не может его изменять.
- Вспомогательная логика, такая как метод получения типа последней записи `last_type_log()`, вынесена в секцию `protected`. Клиентский код не видит этого метода, но классы-наследники, такие как `ExtendedLogManager`, могут его использовать для реализации собственной функциональности (метод `print_last_type`).

### Результаты выполнения работы
Реализованная архитектура успешно демонстрирует концепции «слабого» связывания при строгой инкапсуляции. Тестовый скрипт доказывает, что публичный интерфейс класса `LogManager` стабилен: прямое обращение к внутренним структурам (например, `inspector.vec_logs.front() = ...`) вызывает ошибку компиляции.
